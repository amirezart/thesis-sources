# گردش‌کار ریپو و Git/GitHub

## ۱. همگام‌سازی شروع کار

پیش از هر تغییر مهم:

```bash
git status -sb
git branch --show-current
git remote -v
git fetch --all --prune
git log --oneline --decorate -10
```

سپس `README.md`، `docs/ai-context/PROJECT.md` و سند مرتبط را بخوان. اگر شاخه محلی از remote عقب است، قبل از تغییر وضعیت را روشن کن.

## ۲. محدوده تغییر

- فقط فایل‌های مرتبط با درخواست را تغییر بده.
- تغییرات نامرتبط کاربر را stage یا حذف نکن.
- برای فایل‌های منبع خام، اصل بر عدم ویرایش است؛ اصلاح OCR یا metadata باید مستند باشد.
- فایل جدید باید جایگاه روشن در نقشه ریپو داشته باشد.

## ۳. چرخه تغییر

1. بررسی وضعیت و اسناد مرجع
2. تعریف محدوده و فرض‌ها
3. اعمال تغییر
4. بررسی diff
5. اجرای کنترل کیفیت مرتبط
6. به‌روزرسانی `PROJECT.md` در صورت تغییر تصمیم/وضعیت
7. stage فایل‌های مشخص
8. commit توصیفی
9. push شاخه
10. PR یا merge طبق درخواست
11. verify نهایی روی remote

## ۴. معنای دقیق وضعیت‌ها

- **ویرایش شد:** فایل محلی تغییر کرده است.
- **کامیت شد:** SHA محلی ایجاد و بررسی شده است.
- **پوش شد:** همان SHA روی remote وجود دارد.
- **PR ساخته شد:** شماره/لینک PR موجود است.
- **مرج شد:** PR یا commit در شاخه مقصد ادغام شده است.
- **روی main است:** شاخه `origin/main` شامل commit موردنظر است.

این عبارت‌ها جایگزین هم نیستند.

## ۵. «مرج تو مین»

وقتی کاربر می‌گوید «مرج تو مین»:

- وضعیت و diff را بررسی کن؛
- فقط تغییرات مرتبط را commit کن؛
- شاخه را push کن؛
- تغییر را طبق دسترسی و سیاست ریپو در `main` ادغام کن؛
- `origin/main` را دوباره fetch/pull و وجود commit را verify کن؛
- نام شاخه، SHA، نتیجه بررسی و هر محدودیت را گزارش کن.

اگر دسترسی write/merge وجود ندارد، صریح بگو و patch یا فایل آماده + دستورات دقیق تحویل بده.

## ۶. پیام‌های پیشنهادی

```text
docs: add thesis research protocols
research: add verified notes for Manovich chapter
archive: register reference works and influence links
fix: correct source metadata and page references
chore: update project status and open questions
```

## ۷. حفاظت از داده

- فایل شخصی، اطلاعات تماس، نسخه دارای حق نشر یا داده حساس بدون ضرورت وارد ریپوی عمومی نشود.
- پیش از افزودن PDF/تصویر، حق انتشار و اندازه فایل بررسی شود.
- کلید، توکن، credential و فایل محیطی هرگز commit نشود.

## ۸. گزارش نهایی تغییر

```text

فایل‌ها:
کنترل‌های انجام‌شده:
شاخه/کامیت/PR:
موارد باز یا محدودیت:
```
